## Project Overview

`office-request` ("Office request") is a **Microsoft Power Apps Code App**: a React 19 + TypeScript single-page app, built with Vite, that runs inside the Power Platform and reads/writes **Dataverse** tables through the `@microsoft/power-apps` SDK. The domain is office-supply ordering. It was scaffolded from the standard "React + TypeScript + Vite" template and then wired for Power Apps Code Apps.

Note: `src/App.tsx` is still the default Vite starter (a counter). The Dataverse data layer is fully generated and ready to use, but the actual office-request UI has not been built yet — that's the work this app exists for. The phased build order is in `implementation-plan.md` (starts at Phase 0: bootstrap the Power Apps context in `main.tsx`).

## Product scope & rules

Full detail in `project-scope.md`; keep changes inside these boundaries unless the scope doc is updated first.

**Roles**
- **User** — reads the catalog (available products only), places orders, sees/edits *only their own* orders.
- **Admin** — manages the catalog (add/edit/delete products) and sees/edits/deletes *all* orders.

**Locked decisions**
- **Admin = a Dataverse security role** ("Office Request Admin"). The app reads the current user's roles to toggle admin UI; the role also grants the wider table permissions. Real enforcement is Dataverse, not the React UI — the UI only hides/disables.
- **Own orders** are matched by **`poc_requestedby` = current user**; the app sets `poc_requestedby` on create.
- **Edit window**: users can edit/cancel an order **only while status = `Submitted`**. Once an admin acts on it, it locks for the user (admin can still edit anything).
- **Order status flow**: `Submitted → Approved / Rejected → Assigned → Fulfilled`. Only admins change status.
- **Deleting a product** = soft delete: set **`poc_available = false`** (preserve order history); hard delete only for products never used on an order line.

**Order shape**: an order is a header (`poc_order`) plus one or more lines (`poc_orderline`, each a product + quantity). Build the header, then attach lines via the `poc_Order` / `poc_Product` lookups.

**Out of scope** (do not build without scope sign-off): payments/budgets, inventory tracking, notifications/email, reporting/dashboards, multi-step approval chains, bulk import/export.

## Commands

```bash
npm run dev       # vite dev server (serves the SPA at http://localhost:3000)
npm run build     # tsc -b && vite build — type-check project refs, then bundle to ./dist
npm run lint      # eslint .
npm run preview   # serve the production build locally
```

There is no test runner configured. Talking to live Dataverse data requires an authenticated Power Platform environment; the binding (app id, environment id, data sources) lives in `power.config.json`, and the app is run/published through the Power Platform CLI (`pac`).

## Architecture

### Data layer — `src/generated/` (autogenerated; never hand-edit)

Every file here carries a "This file is autogenerated. Do not edit this file directly." header. It is regenerated from the Dataverse schema by the Power Platform tooling, so any manual change is lost. Application code imports from here and calls the service classes — it never calls the SDK directly.

Per Dataverse table there is a model file and a service file, re-exported through `src/generated/index.ts`:

- **`models/Poc_<table>Model.ts`** — two interfaces per table: `Poc_<table>Base` (writable columns, used for create/update) and `Poc_<table>` (extends Base with system/read-only fields like `createdon`, `*name` display values, and `_<lookup>_value`). Choice (option-set) columns are emitted as `as const` maps plus a key type, e.g. order status `{ Submitted, Approved, Assigned, Fulfilled, Rejected }` and product category `{ Peripherals, Monitors, Laptops, Accessories, Furniture, Other }`.
- **`models/CommonModels.ts`** — shared query option types `IGetOptions` (`select`) and `IGetAllOptions` (`filter`, `orderBy`, `top`, `skip`, paging).
- **`services/Poc_<table>Service.ts`** — a static class (`create`, `update`, `delete`, `get`, `getAll`, `getMetadata`) backed by `getClient(dataSourcesInfo)` from `@microsoft/power-apps/data`, where `dataSourcesInfo` is imported from `.power/schemas/appschemas/dataSourcesInfo`.

### Domain model (three Dataverse tables, publisher prefix `poc_`)

An ordering flow: **products** (catalog: name, category, unit price, availability) → **orders** (header: requested-by, requested-date, status, notes) → **orderlines** (line items: quantity, plus lookups `poc_Order` and `poc_Product` set via `"poc_Order@odata.bind"` / `"poc_Product@odata.bind"`). Friendly data-source names are `products` / `orders` / `orderlines` (see `power.config.json`).

### App shell

`index.html` → `src/main.tsx` (React root, `StrictMode`) → `src/App.tsx` (UI). Styling: `src/App.css`, `src/index.css`. Static assets in `src/assets/`; `public/` is served as-is.

### Power Apps SDK runtime (`@microsoft/power-apps`)

Verified against the installed types (v1.1.x) — easy to get wrong, so confirm here before guessing:

- **App context** lives in `@microsoft/power-apps/app`: `setConfig(...)` (optional logger) and `getContext(): Promise<IContext>`. There is **no** `initialize` export and **no** `PowerProvider` component. `main.tsx` does not call this yet (Phase 0).
- **Current user** comes from `getContext()` → `IContext.user` (`objectId`, `fullName`, `userPrincipalName`, `tenantId`). Use `user.objectId` (Entra object id, stable) as the identity key — it is **not** the Dataverse `systemuserid`, and the data client has **no** `getUserId()`. Because `poc_requestedby` is a free-text string, store `objectId` there on create and filter "my orders" by the same value.
- **Data + custom requests**: `getClient(dataSourcesInfo)` (from `@microsoft/power-apps/data`) returns a `DataClient` with `createRecordAsync` / `updateRecordAsync` / `deleteRecordAsync` / `retrieveRecordAsync` / `retrieveMultipleRecordsAsync` / `executeAsync` (plus file/image helpers). There is **no** roles API, so the "Office Request Admin" check must go through `executeAsync` (a Dataverse query) or an added role data source — this is the main open unknown (see `implementation-plan.md`, Phase 1).

### Config

- `power.config.json` — Power Platform Code App binding: `appId`, `environmentId`, `buildPath`/`buildEntryPoint`, `localAppUrl`, and the Dataverse `dataSources`.
- `vite.config.ts` — Vite with `@vitejs/plugin-react` and the `powerApps()` plugin from `@microsoft/power-apps-vite/plugin`.
- TypeScript project references: `tsconfig.json` (solution) → `tsconfig.app.json` (app, `strict`, bundler resolution, `noEmit`, `react-jsx`, `verbatimModuleSyntax`, `noUnusedLocals`/`noUnusedParameters`) and `tsconfig.node.json` (Vite/Node tooling). `npm run build` runs `tsc -b`, so a type error in **either** referenced project fails the build before bundling.
- `eslint.config.js` — flat config with `typescript-eslint` plus the React Hooks and React Refresh plugins.

## Conventions

- Treat `src/generated/**` and `.power/**` as build artifacts — regenerate via the Power Platform CLI when the schema/data sources change rather than editing by hand.
- To consume data, import the service classes and model types from `src/generated` (e.g. `Poc_productsService.getAll(...)`) and use the `Base` types for writes, the full types for reads.
- When adding a new table, add the data source through the Power Platform tooling so its model + service are generated, then import them from `src/generated`.
